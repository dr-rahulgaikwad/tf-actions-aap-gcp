# Memo

Guide to writing memos for quick decisions and proposals at HashiCorp.

## What is a Memo?

A **Memo** is a lightweight document for quick decisions, impact assessments, and proposals that don't require the full structure of an RFC or PRD. Memos enable fast alignment on small-scope topics while preserving context.

Memos are managed by Hermes (HashiCorp's document management system) and follow a standard format with auto-generated headers.

## When to Use a Memo

### Use Memo For:

- **Quick decisions**: Small-scope choices needing stakeholder input
- **Impact assessments**: Evaluating potential changes or initiatives
- **Proposals**: Suggesting alternatives or approaches
- **R&D evaluations**: Technical or product evaluations
- **Process changes**: Updates to workflows or policies
- **Trade-off analysis**: Comparing options for a decision
- **Feedback requests**: Seeking input on a specific topic

### Don't Use Memo For:

- **Technical implementation**: Use RFC for detailed architecture and design
- **Product requirements**: Use PRD for feature specifications and acceptance criteria
- **Product vision**: Use PRFAQ for customer-centric narratives
- **Large-scope changes**: Use RFC or PRD for multi-week initiatives
- **Bug reports**: Use GitHub issues for defects
- **Quick questions**: Use Slack for simple clarifications

### Decision Matrix

```
Scope of Change:

Small (< 1 week)
  └─ Memo or direct discussion

Medium (1-4 weeks, single component)
  └─ RFC (if technical) or PRD (if product)

Large (> 1 month, cross-team)
  ├─ PRFAQ (if customer-facing vision)
  ├─ PRD (if product requirements)
  └─ RFC (if technical implementation)

Just need opinions?
  └─ Memo (then potentially escalate to RFC/PRD)
```

## Memo Structure

### Standard Format

```
| [Memo] ABC-123: Title of Memo |  |
| :---- | :---- |
| **Summary:** Brief description |  |
|  |  |
| **Created**: Date | **Status**: WIP | In-Review | Approved | Obsolete |
|  |  |
| **Product**: Product name | **Owner**: owner@hashicorp.com |
| **Tags**: N/A | **Contributors**: N/A |
|  | **Approvers**: N/A |
|  |  |
| **NOTE**: This document is managed by Hermes and this header will be periodically overwritten using document metadata. |  |

**Background**
Describe the situation

**R&D Evaluation | Impact Assessment**
Explain the challenges

**Proposed Solution**
Offer an alternative solution

**Next Steps**
Ask for feedback
```

### 1. Header Table

**Auto-generated by Hermes** based on document metadata.

**Fields**:
- **Title**: `[Memo] ABC-123: Title`
  - ABC-123 is the Hermes document ID (auto-assigned)
  - Title should be concise and descriptive
- **Summary**: One sentence describing the memo
- **Created**: Date (auto-populated)
- **Status**: WIP → In-Review → Approved → Obsolete
- **Product**: Product or team this affects
- **Owner**: Author of the memo
- **Tags**: Optional categorization
- **Contributors**: Co-authors or collaborators
- **Approvers**: Stakeholders who should review

**Important**: Hermes overwrites this header periodically. Edit metadata in Hermes, not directly in the document.

### 2. Background

**Purpose**: Describe the current situation and context.

**Guidelines**:
- **Concise**: 1-3 paragraphs usually sufficient
- **Set context**: What's happening now?
- **Explain relevance**: Why does this matter?
- **Link to evidence**: Data, Slack threads, GitHub issues
- **Assume reader is new**: Provide enough context to understand

**Example**:
```
**Background**

Our current deployment process for HCP Vault requires manual approval
from the SRE team for each production change. This approval step typically
takes 4-8 hours during business hours and up to 24 hours during weekends
or holidays.

Recent incidents (INC-456, INC-789) were delayed due to waiting for SRE
approval. Engineering teams have requested faster deployment cycles to
iterate on customer feedback more quickly.

The SRE team is concerned about removing approval gates due to risk of
misconfigurations or security issues in production deployments.
```

### 3. R&D Evaluation | Impact Assessment

**Purpose**: Explain the challenges, trade-offs, or analysis.

**Guidelines**:
- **Identify challenges**: What are the blockers or concerns?
- **Analyze impact**: Who/what is affected?
- **Consider trade-offs**: Pros and cons of different approaches
- **Quantify when possible**: Time saved, cost reduced, risk increased
- **Be objective**: Present multiple perspectives

**Example**:
```
**Impact Assessment**

**Current State**:
- Average deployment time: 6 hours (2 hours engineering + 4 hours approval)
- Deployments per week: 3-5
- Deployment errors (last 6 months): 2 incidents, both caught in staging

**If we remove approval gate**:
- Pros:
  - Reduce deployment time to 2 hours (4-hour savings per deployment)
  - Enable weekend deployments without SRE on-call
  - Faster iteration on customer feedback
  - Engineering velocity improvement: ~12 hours/week saved

- Cons:
  - Risk of misconfiguration reaching production
  - SRE team loses visibility into production changes
  - No safety net for junior engineers' deployments

**If we automate approval checks**:
- Pros:
  - Maintain safety checks (policy validation, security scans)
  - Fast approval for compliant changes (<10 minutes)
  - Audit trail preserved
  - SRE team alerted to policy violations only

- Cons:
  - Engineering effort: ~2 weeks to build automation
  - Policy definitions need to be comprehensive
  - Edge cases may require manual override workflow
```

### 4. Proposed Solution

**Purpose**: Recommend a specific approach or alternative.

**Guidelines**:
- **Be specific**: Concrete proposal, not just analysis
- **Explain rationale**: Why this solution over alternatives?
- **Outline steps**: What needs to happen?
- **Identify owners**: Who should execute this?
- **Set timeline**: When should this be done?

**Example**:
```
**Proposed Solution**

Implement automated deployment approval using policy-as-code:

1. Define deployment policies (Week 1)
   - Security scan must pass
   - Configuration schema validation
   - No changes to production during freeze windows
   - Rollback plan required
   Owner: SRE team

2. Build automation (Week 2-3)
   - GitHub Actions workflow runs policy checks
   - Auto-approve if all policies pass
   - Alert SRE team if policies fail
   - Manual override option for emergencies
   Owner: Platform Engineering team

3. Pilot with one service (Week 4)
   - Test with HCP Vault Control Plane
   - Gather feedback from eng and SRE
   - Refine policies based on learnings
   Owner: HCP Vault team + SRE

4. Rollout to all services (Week 5-6)
   - Gradual rollout to other HCP services
   - Monitor for policy violations
   - Iterate on policies as needed
   Owner: Platform Engineering + SRE

This approach maintains safety while reducing approval time from 4 hours
to <10 minutes, saving ~12 hours per week of engineering time.
```

### 5. Next Steps

**Purpose**: Request feedback and define actions.

**Guidelines**:
- **Request feedback**: What do you need from readers?
- **Set deadline**: When do you need responses?
- **Propose actions**: What should happen next?
- **Identify decision makers**: Who needs to approve?
- **Schedule follow-up**: Meeting or async decision?

**Example**:
```
**Next Steps**

Please review this proposal and provide feedback by **Friday, Jan 20**.

**Questions for stakeholders**:
- SRE team: Are the proposed policies sufficient for safety?
- Engineering teams: Would this solution meet your deployment speed needs?
- Security team: Any security concerns with automated approval?

**Next steps**:
1. Gather feedback (by Jan 20)
2. Schedule sync meeting if needed (week of Jan 23)
3. Update memo based on feedback
4. Get approval from SRE Lead and VP Engineering
5. Create RFC for technical implementation (if approved)

**Decision makers**: Sarah Chen (SRE Lead), John Doe (VP Engineering)
```

## Memo Style Guidelines

### Beautiful is Better

Keep memos clean and well-organized. Use consistent formatting.

### Formatting

- **Headings**: Bold headings for main sections
- **Lists**: Use bullets or numbered lists for clarity
- **Tables**: Compare options side-by-side
- **Bold key terms**: Emphasize important points
- **Links**: Reference supporting documents

### Length

- **Concise**: 1-3 pages maximum
- **Focused**: Single topic or decision
- **Skimmable**: Busy readers should grasp main points quickly

### Tone

- **Objective**: Present facts and analysis
- **Neutral**: Don't advocate too strongly
- **Constructive**: Focus on solutions, not just problems
- **Respectful**: Acknowledge different perspectives

## Memo Workflow

### 1. Creating a Memo

```
1. Create document in Hermes
   ├─ Use Memo template
   └─ Hermes assigns document ID (ABC-123)

2. Fill in metadata (in Hermes)
   ├─ Title
   ├─ Summary
   ├─ Owner
   ├─ Product/Team
   ├─ Approvers (if needed)
   └─ Status: WIP

3. Write memo content
   ├─ Background
   ├─ Impact Assessment or R&D Evaluation
   ├─ Proposed Solution
   └─ Next Steps

4. Set status to "In-Review"
   └─ Share with stakeholders
```

### 2. Sharing for Feedback

```
1. Share link in Slack
   ├─ Post in relevant channel (#engineering, #product)
   ├─ Tag key stakeholders
   └─ Set expectation for review timeline

2. Enable comments in Google Docs
   └─ Reviewers can comment inline

3. Request feedback
   └─ Specify what questions you need answered
```

### 3. Incorporating Feedback

```
1. Address comments
   ├─ Update memo based on feedback
   └─ Respond to comments

2. Iterate as needed
   └─ May require multiple rounds

3. Reach consensus
   └─ Get agreement from decision makers
```

### 4. Finalizing

```
1. Get approval (if required)
   ├─ Approvers listed in header
   └─ Update status to "Approved"

2. Document decision
   └─ What was decided?

3. Execute next steps
   ├─ Create RFC or PRD if needed
   ├─ Create GitHub issues
   └─ Update relevant documentation

4. Archive when complete
   └─ Set status to "Obsolete" when superseded
```

## Common Memo Types

### Impact Assessment Memo

**When**: Evaluating potential changes or initiatives.

**Focus**:
- What's the impact of this change?
- Who is affected?
- What are the trade-offs?
- Should we proceed?

**Example Topics**:
- Deprecating a feature
- Changing a pricing model
- Reorganizing teams
- Adopting a new technology

### Decision Memo

**When**: Need to make a choice between options.

**Focus**:
- What are the options?
- What are pros/cons of each?
- What's the recommendation?
- Who needs to approve?

**Example Topics**:
- Choosing between technologies
- Prioritizing features
- Resolving technical debt vs. new features
- Selecting vendors

### Proposal Memo

**When**: Suggesting a new approach or initiative.

**Focus**:
- What's the problem?
- What's the proposed solution?
- What's required to execute?
- What's the ask?

**Example Topics**:
- Process improvements
- New team structures
- Infrastructure changes
- Policy updates

### R&D Evaluation Memo

**When**: Researching technologies or approaches.

**Focus**:
- What did we evaluate?
- What are the findings?
- What's the recommendation?
- What are next steps?

**Example Topics**:
- Technology spike results
- Proof of concept findings
- Competitive analysis
- Market research

## Best Practices

### For Memo Authors

1. **Be concise**: Get to the point quickly
2. **Provide context**: Background section is important
3. **Propose a solution**: Don't just describe problems
4. **Request specific feedback**: What do you need from readers?
5. **Set deadlines**: When do you need responses?
6. **Use Hermes**: Track status and approvals properly
7. **Link to evidence**: Data, Slack threads, documents
8. **Update based on feedback**: Iterate until consensus

### For Memo Reviewers

1. **Read background first**: Understand context
2. **Comment inline**: Use Google Docs comments
3. **Be specific**: "Approve" or "concerns with X"
4. **Respond by deadline**: Author is waiting for feedback
5. **Ask questions**: Clarify before approving
6. **Focus on substance**: Not minor formatting issues
7. **Suggest alternatives**: If you disagree with proposal

### Common Pitfalls

**❌ Too long**: Memos should be 1-3 pages, not 10 pages
- Fix: Consider RFC or PRD instead

**❌ No clear ask**: Readers don't know what feedback you need
- Fix: Add explicit questions in Next Steps

**❌ Missing context**: Assumes readers know background
- Fix: Expand Background section

**❌ Analysis without recommendation**: Just presents options
- Fix: Add Proposed Solution with specific recommendation

**❌ No timeline**: Unclear when decision is needed
- Fix: Set deadline in Next Steps

## When to Escalate from Memo

### Memo → RFC

**Escalate when**:
- Proposal requires detailed technical design
- Multiple engineering teams affected
- Significant architecture changes
- Implementation is complex

**Example**: Memo proposes new caching strategy → RFC details cache design, API changes, migration plan

### Memo → PRD

**Escalate when**:
- Proposal becomes a feature requiring product requirements
- Need to define acceptance criteria and phased delivery
- Requires user research and persona definition
- Customer-facing changes

**Example**: Memo proposes improving widget publishing → PRD defines requirements, user research, acceptance criteria

### Memo → PRFAQ

**Escalate when**:
- Proposal becomes a major customer-facing capability
- Need to define vision and customer narrative
- Requires leadership buy-in on strategy
- Market positioning is important

**Example**: Memo proposes secrets detection feature → PRFAQ describes customer value, press release, business case

## Troubleshooting

### Not Getting Feedback

**Problem**: Shared memo but no one is reviewing.

**Solutions**:
1. Tag specific people in Slack post
2. Set clear deadline: "Need feedback by Friday"
3. DM key stakeholders directly
4. Schedule sync meeting to discuss
5. Follow up via email if urgent

### Scope Creeping

**Problem**: Memo keeps growing and becoming complex.

**Solutions**:
1. Stop and create RFC or PRD instead
2. Split into multiple memos (narrow each one)
3. Move details to appendix, keep main memo concise
4. Link to separate documents for details

### Stakeholders Disagree

**Problem**: Reviewers have conflicting opinions.

**Solutions**:
1. Schedule meeting to align
2. Present options and let decision maker choose
3. Compromise: combine elements of different approaches
4. Escalate to higher leadership if needed
5. Document disagreement and chosen path

### Decision Not Clear

**Problem**: After review, still unclear what to do next.

**Solutions**:
1. Explicitly document the decision in memo
2. Update Next Steps with concrete actions
3. Assign owners to each action item
4. Set timeline for execution
5. Schedule follow-up check-in
