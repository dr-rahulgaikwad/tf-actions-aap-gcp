# UXDR-014: TFC Integration UX Decisions

**Type**: UX Decision Record (UXDR)
**Related**: 010.Invisible-User-Journey, 013.TFC-Integration-Requirements

---

## Decision Log

### 2025-01-15: Establish "Invisible First" UX Direction

**Context**: The 013.TFC-Integration-Requirements document specifies comprehensive UI additions to HCP Terraform Cloud. Upon review against the "invisible agent" paradigm (010), we identified a tension: the proposed UI creates substantial surfaces that users must monitor, contradicting the goal of delivering value where engineers already work (GitHub).

**Decision**: Prioritize an **invisible experience**:
- TFC UI is for **configuration and exception handling**, not monitoring
- GitHub remains the **primary interface** for scan results and remediation
- Success states are **minimal**; only problems demand attention
- Users **trust the automation** rather than manually triggering actions

**Changes to 013**:

| Priority | Change | Rationale |
|----------|--------|-----------|
| P0 | Add Notification Preferences section | Users can only "set and forget" if they trust they'll be alerted when action is needed |
| P0 | Specify deep link query parameters (`iac_scanning`, `status`, `error_code`) | Required for OAuth return flow |
| P1 | Collapse Agent Activity Dashboard to single-line health summary | Default state should confirm health without demanding attention |
| P1 | Move "Scan Now" to overflow menu | Prominent manual triggers imply automation isn't trusted |
| P1 | Simplify scan status to 3 states (Running/Success/Failed) | Detailed stages (Cloning/Scanning/Posting) are implementation noise |
| P1 | Project view: lead with "Needs Attention", collapse clean workspaces | Users need to know what requires action, not reassurance that scans ran |
| P1 | Add explicit "Clean State" mockup | Most common state needs explicit minimal design |
| P1 | Add "First-Scan Pending" state | Trust-building moment between enable and first results |
| P2 | Clarify link destinations with ↗ for external (GitHub) links | Reduce confusion about where actions navigate |
| P2 | Remove agent names from default status view | Implementation detail; available in debug mode |

**Consequences**:
- (+) Reduced cognitive load; users don't monitor TFC
- (+) Clearer value prop: "Enable once, value in GitHub"
- (-) Less dashboard visibility for stakeholders wanting reports
- (-) Extra clicks when debugging (detail is collapsed)

**Mitigations**: Last-scan timestamp always visible; expanded view persists per-session.

---

## Guiding Principles

These principles inform all future UX decisions for TFC integration:

1. **Silence is Success** — When working, the UI is nearly invisible
2. **GitHub is Primary** — TFC links to GitHub; doesn't duplicate it
3. **Automation Over Manual** — De-emphasize manual triggers
4. **Progressive Disclosure** — Details available on-demand, not by default
5. **Notifications Enable Invisibility** — Alert config is P0, not afterthought
6. **Attention-Required Over Status** — Lead with problems, not comprehensive stats

## Future Decisions

### 2025-01-15: Start Minimal, Earn Complexity

**Context**: Compared TFC integration to IAC remediation tool, which shows detailed agent activity. Question: should TFC have similar depth?

**Decision**: No. Start invisible-first. Don't build debugging/monitoring UI upfront.

**Rationale**:
- We don't yet know which agent issues will actually occur in production
- Building rich debugging UI now means guessing at problems we haven't seen
- Every UI surface adds product scope, support burden, and maintenance cost
- GitHub already shows results (Check Runs, PR comments)—TFC doesn't need to duplicate

**Approach**:
- Ship minimal collapsed view
- When agents fail in production, observe what users actually need
- Add transparency incrementally based on real support patterns
- Let real-world usage reveal which visibility is worth the cost

**Trade-off accepted**: Early adopters may want more visibility. We'll add it when demand proves real—not hypothetical.
